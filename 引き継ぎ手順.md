## 1. 常駐化の仕組み

`起動.bat` を対話コンソールで実行する運用では、実行者がログオフ／ユーザー切替すると
`node server.js` プロセスが終了し、サイトが停止する。同じサーバーPC上で他サイトの
改修作業のためにログインし直すと、HILSが巻き添えで停止していたのはこれが原因である。

対策として、タスクスケジューラに「ログオン有無に関わらず実行」の常駐タスクを登録する。
これにより、誰もログインしていない状態でも `node server.js` が稼働し続ける。

Windows の仕様上、資格情報を保存した常駐実行にはユーザーアカウントのパスワードが
必要である（SYSTEM実行にすると、SharePointプロキシ認証（NTLM）がマシンアカウントで
行われてしまい失敗するため不可）。したがって、常駐タスクは特定の人物のアカウントに
紐づく。この人物が変わる場合は「5. 権限引き継ぎ手順」を実施する。

また、タスクスケジューラの「ログオン有無に関わらず実行」セッションでは、対話ログイン時と異なり、
現在ユーザーの暗黙の資格情報によるNTLM SSOが社内プロキシに拒否される事象が確認されている
（`sp_helper.py`が407 Proxy Authentication Requiredで失敗する）。このため`サービス登録.ps1`は、
常駐タスクの登録と同時に、同じパスワードをWindows資格情報マネージャー
（`cmdkey`、ターゲット名`HILS_PROXY_AUTH`）にも登録し、`sp_helper.py`が明示的な資格情報で
NTLM認証できるようにしている。

## 2. 初回セットアップ

1. 管理者権限のPowerShellで、実行アカウント自身が対話ログインして `サービス登録.ps1` を実行する
   （`cmdkey`への資格情報登録は実行者自身のユーザーに対して行われるため、代理実行はできない）。
2. 実行アカウント（既定は現在ログイン中のアカウント）とパスワードを入力する。
   常駐タスクの登録に加え、資格情報マネージャー（`HILS_PROXY_AUTH`）への登録も自動で行われる。
3. 登録完了後、`schtasks /Run /TN "HILS予約サイト"` で即時起動し、動作確認する
   （`http://localhost:3000/` および他PCから `http://172.25.7.82:3000/`）。
4. ファイアウォールでTCP 3000番の受信が許可されていない場合は以下を実行する。
   ```
   netsh advfirewall firewall add rule name="HILS予約サイト 3000" dir=in action=allow protocol=TCP localport=3000
   ```

## 3. 日常運用コマンド

| 操作 | コマンド |
|---|---|
| 状態確認 | `schtasks /Query /TN "HILS予約サイト" /V` |
| 即時起動 | `schtasks /Run /TN "HILS予約サイト"` |
| 停止 | `schtasks /End /TN "HILS予約サイト"` |
| タスク削除 | `Unregister-ScheduledTask -TaskName "HILS予約サイト" -Confirm:$false`（PowerShell） |

事務局モードでの画面確認は `http://<サーバーIP>:3000/?user=admin` を使用する。

## 4. パスワード変更時の手順（最後の変更から90日以内に変更必須）

実行アカウントのパスワードを変更したら、常駐タスクの資格情報も更新しないと
タスクが起動しなくなる。以下を都度実施する。

1. `サービス登録.ps1` を再実行し、同じアカウント・新しいパスワードで再登録する
   （常駐タスクの資格情報と、資格情報マネージャー`HILS_PROXY_AUTH`の両方が新パスワードに更新される）。
2. `schtasks /Run /TN "HILS予約サイト"` で起動確認する。
3. サイトが表示されるがSharePointの予約データが取得できない場合、Azure ADの
   リフレッシュトークンも失効している可能性がある。`再サインイン.bat` を実行し、
   表示されたデバイスコードでブラウザからMicrosoftアカウントに再サインインする。

## 5. 権限引き継ぎ手順（担当者変更時）

常駐タスクの実行アカウント、SharePoint/Graphの認証、メール送信元は
いずれも「対話ログインで最初にセットアップした担当者個人」に紐づいている。
担当者が変わる際は、以下の3点をすべて新担当者のものへ切り替える。

1. **常駐タスクの実行アカウント差し替え**
   新担当者自身が対話ログインし、`サービス登録.ps1` を実行して新担当者のドメインアカウント・
   パスワードを入力して再登録する（資格情報マネージャー`HILS_PROXY_AUTH`も新担当者自身の
   ユーザーに対して同時に登録される。旧担当者が代理実行しても新担当者の資格情報マネージャーには
   登録されないため、新担当者自身の実行が必須）。

2. **SharePoint/Graph認証の切り替え**
   新担当者のアカウントで `再サインイン.bat` を実行し、表示されるデバイスコードで
   新担当者自身のMicrosoftアカウントからブラウザでサインインする。
   これにより `backend\token_cache.bin` が新担当者のトークンに更新される
   （旧担当者のトークンは上書きされ、以後は使用されない）。

3. **メール送信元の変更**
   `backend\.env` の `MAIL_FROM` を新担当者のメールアドレスに変更する。
   Graph APIのメール送信はサインイン中のアカウント名義（`/me/sendMail`）で行われるため、
   `MAIL_FROM` を実際のサインインアカウントと一致させておく必要がある。
   **他者にメールが送信される設定変更のため、変更前に関係者へ確認を取ること。**

4. 上記3点を実施後、動作確認（サイト表示・予約データ取得・メール送信テスト）を行う。

## 6. トラブルシューティング

| 症状 | 確認すること |
|---|---|
| サイトにアクセスできない | `schtasks /Query /TN "HILS予約サイト" /V` でタスクの状態・最終実行結果を確認。タスクマネージャーで `node.exe` プロセスの有無を確認 |
| タスクが起動しない（実行結果コードが `0x8007052…` 系） | パスワード変更後、タスクの保存パスワードが古い可能性。「4. パスワード変更時の手順」を実施 |
| サイトは表示されるが予約一覧が空・エラー | MSALトークン失効の可能性。`再サインイン.bat` を実行 |
| 社外（プロキシ経由）に繋がらない、`daemon.log`に`407 Proxy Authentication Required`が出る | `.env` の `PROXY_URL` が有効か確認。加えて `cmdkey /list:HILS_PROXY_AUTH` で資格情報マネージャーへの登録有無を確認し、無ければ「2. 初回セットアップ」を実行アカウント自身で再実行する |
| 他PCから `http://<サーバーIP>:3000/` に繋がらない（localhostは繋がる） | サーバーPCのファイアウォールでTCP 3000番の受信が許可されているか確認 |
| ポート3000が使用中でタスクが即終了する | 開発起動時に起動した旧プロセス（`node server.js`／Viteの開発サーバー）が残っていないか確認し、終了させる |
